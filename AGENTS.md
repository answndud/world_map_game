# AGENTS.md

## 목적

이 저장소는 AI 에이전트의 도움을 받아 개발하되, 최종적으로는 사용자가 설계와 구현을 직접 설명할 수 있어야 한다.

즉, 모든 작업은 아래 두 가지를 동시에 만족해야 한다.

- 기능이 실제로 동작할 것
- 사용자가 면접과 포트폴리오에서 그 기능을 설명할 수 있을 만큼 문서가 남을 것

## 기본 원칙

- 이 프로젝트는 `서버 주도 게임 플랫폼`이다.
- 프론트는 입력과 표현을 담당하고, 게임 상태와 정답 판정은 백엔드가 책임진다.
- 한 번에 큰 기능 여러 개를 밀어 넣지 말고, `작게 완결된 단위`로 구현한다.
- 구현 속도보다 `설명 가능한 구조`를 우선한다.
- 새 기능을 만들 때마다 "왜 이렇게 설계했는가"를 반드시 남긴다.

## 작업 시작 전 절차

모든 에이전트는 작업 전에 아래 순서를 따른다.

1. [README.md](/Users/alex/project/worldmap/README.md)를 읽고 현재 프로젝트 방향을 확인한다.
2. [docs/PORTFOLIO_PLAYBOOK.md](/Users/alex/project/worldmap/docs/PORTFOLIO_PLAYBOOK.md)에서 현재 단계와 다음 목표를 확인한다.
3. 이번 작업이 어느 단계에 속하는지 명시한다.
4. 한 번에 구현할 범위를 `하나의 작은 기능 조각`으로 제한한다.

예시:

- "위치 찾기 게임 전체"가 아니라 `위치 찾기 세션 시작 API`
- "랭킹 시스템 전체"가 아니라 `게임 종료 후 Redis Sorted Set 반영`

## 구현 순서 규칙

기능은 반드시 아래 흐름을 우선한다.

1. 프로젝트 뼈대
2. 국가 데이터와 시드
3. 위치 찾기 게임 Level 1
4. 인구수 맞추기 게임 Level 1
5. Redis 랭킹
6. 설문 기반 추천 로직
7. LLM 설명 기능
8. 인증, 전적, 마이페이지
9. Level 2 고도화와 실시간 전달 개선

이 순서를 크게 벗어나는 구현은 이유를 문서에 남겨야 한다.

## 모든 작업의 필수 산출물

의미 있는 코드 변경이 있었다면, 에이전트는 반드시 아래를 같이 남긴다.

### 1. 코드

- 실제 동작하는 구현
- 필요한 테스트

### 2. 플레이북 업데이트

[docs/PORTFOLIO_PLAYBOOK.md](/Users/alex/project/worldmap/docs/PORTFOLIO_PLAYBOOK.md)의 해당 단계에 아래 내용을 반영한다.

- 진행 상태
- 이번 작업이 왜 필요한지
- 다음에 이어서 할 일

### 3. 작업 로그 업데이트

[docs/WORKLOG.md](/Users/alex/project/worldmap/docs/WORKLOG.md)에 아래 항목을 기록한다.

- 날짜
- 현재 단계
- 구현한 기능
- 변경 파일
- 요청 흐름 / 데이터 흐름
- 핵심 도메인 개념
- 예외 상황 또는 엣지 케이스
- 테스트 내용
- 면접에서 30초 안에 설명하는 요약
- 아직 내가 이해가 부족한 부분

### 4. 블로그 글 업데이트

`설계 이유`, `요청 흐름`, `도메인 모델`, `테스트`를 설명할 수 있는 기능 조각이라면 같은 턴에 `blog/`도 같이 업데이트한다.

기본 원칙:

- API 추가
- 도메인 구조 변경
- 게임 루프 변경
- Redis / 랭킹 / 추천 / LLM 연결
- 테스트가 붙은 의미 있는 기능 완료

위 항목은 `blog/NN-*.md` 글 작성 또는 기존 글 업데이트를 기본값으로 본다.

예외:

- 버튼 문구 수정
- 단순 CSS 미세 조정
- 오타 수정
- 테스트와 설계 설명 가치가 거의 없는 작은 버그 수정

이런 작은 변경은 블로그를 생략할 수 있다. 다만 생략했다면 왜 생략했는지 결과 설명이나 작업 로그에 짧게 남긴다.

문서 동기화가 필요한 경우, 프로젝트 로컬 스킬 [`$worldmap-doc-sync`](/Users/alex/project/worldmap/.agents/skills/worldmap-doc-sync/SKILL.md)을 명시적으로 사용한다.

## 답변 규칙

에이전트는 작업 결과를 설명할 때 아래 내용을 우선 포함한다.

- 무엇을 만들었는가
- 왜 지금 이 기능을 만들었는가
- 어디에서 요청이 시작되어 어디서 상태가 바뀌는가
- 왜 이 로직이 컨트롤러가 아니라 서비스 또는 도메인에 있어야 하는가
- 무엇을 테스트했는가
- 다음 단계는 무엇인가

## 금지 사항

- 여러 개의 대형 기능을 한 번에 구현하지 않는다.
- 사용자가 설명하기 어려운 추상화나 프레임워크를 불필요하게 추가하지 않는다.
- 프론트에서 처리하면 편하다는 이유로 서버 책임을 넘기지 않는다.
- 문서 업데이트 없이 코드만 수정하지 않는다.
- "작동한다"만 남기고, 설계 이유를 생략하지 않는다.

## 단계 종료 기준

각 단계는 아래 조건을 만족해야 다음 단계로 넘어간다.

- 핵심 기능이 동작한다.
- 최소 1개 이상의 핵심 테스트가 있다.
- [docs/WORKLOG.md](/Users/alex/project/worldmap/docs/WORKLOG.md)에 기록이 남아 있다.
- 설명 가치가 있는 기능 조각이면 `blog/`에도 대응 글이나 업데이트가 남아 있다.
- 사용자가 아래 질문에 답할 수 있다.
  - 왜 이 기능이 필요한가?
  - 요청은 어떤 흐름으로 처리되는가?
  - 핵심 엔티티와 상태 변화는 무엇인가?
  - 실패 케이스는 무엇인가?
  - 왜 이런 기술을 선택했는가?

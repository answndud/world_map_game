# Blog Rebuild Guide

## 1. 목표

WorldMap 블로그를 "설명용 글"이 아니라 "재현형 기술 시리즈"로 유지하는 것이 목표입니다.
여기서 `재현형`은 문서만 떼어 놓고 읽어도 된다는 뜻이 아니라, 문서가 독자를 현재 저장소의 실제 코드, 테스트, 설정 파일로 정확히 안내해야 한다는 뜻입니다.

좋은 글의 기준은 아래입니다.

- 독자가 먼저 열 파일을 결정할 수 있다
- 독자가 구현 순서를 따라갈 수 있다
- 독자가 테스트와 검증 명령을 그대로 실행할 수 있다
- 독자가 면접에서 왜 이렇게 설계했는지 답할 수 있다

이 문서는 그 기준을 지키기 위한 공통 규칙입니다.

## 2. 각 글에 반드시 들어갈 섹션

본편 18개는 아래 순서를 **최소 골격**으로 유지합니다.

중요한 점은 이 19개 섹션이 “최대 개수”가 아니라는 것입니다.
특히 `16~18`처럼 runtime / hardening / verification을 다루는 후반부 대표 글은
아래 골격을 지킨 뒤에도 표, 테스트 맵, 운영 순서, 실패 분류를 위해 섹션을 더 늘릴 수 있습니다.

즉 기준은 이렇게 이해하는 편이 맞습니다.

- `19개 섹션` = 반드시 포함해야 하는 최소 teaching skeleton
- `19개 초과` = 설명 밀도를 위해 허용되는 확장

섹션 수를 억지로 맞추느라 설명이 약해지면 안 됩니다.

### 1. `이번 글에서 풀 문제`

- 이번 글이 닫는 문제를 한 문장으로 먼저 적습니다.
- "무엇을 만들었다"보다 "무슨 혼란을 줄였는가"가 먼저 나와야 합니다.

### 2. `최종 도착 상태`

- 이 글이 끝났을 때 어떤 화면, API, 상태 전이가 살아 있어야 하는지 적습니다.
- 독자가 "여기까지 구현하면 무엇이 보이는가"를 먼저 이해하게 합니다.

### 3. `먼저 알아둘 개념`

- 글에서 반복 사용할 핵심 용어를 미리 풉니다.
- 초심자가 바로 용어 벽에 막히지 않게 합니다.

### 4. `이번 글에서 다룰 파일`

- 실제 파일 경로를 적습니다.
- 클래스, 설정 파일, 템플릿, JS, 테스트 파일을 빠짐없이 연결합니다.

### 5. `핵심 도메인 모델 / 상태`

- 이 글의 핵심 엔티티, value, 상태 전이를 적습니다.
- 예: `Session / Stage / Attempt`, `guestSessionKey`, `LeaderboardRecord`, `RecommendationFeedbackContext`

### 6. `설계 구상`

- 왜 이 경계를 서비스로 뒀는지, 왜 컨트롤러가 아닌지, 왜 별도 read model이 필요한지 설명합니다.

### 7. `코드 설명`

- 중요한 클래스와 메서드를 역할별로 설명합니다.
- 가능하면 "어디서 상태가 바뀌는가"를 같이 적습니다.

### 8. `요청 흐름 / 상태 변화`

- 텍스트 플로우나 코드 블록으로 요청 시작점과 상태 변경 지점을 적습니다.
- 이 시리즈에서는 "요청이 어디서 시작돼 어디서 상태가 바뀌는가"가 가장 중요합니다.

### 9. `실패 케이스 / 예외 처리`

- 이 섹션이 없으면 글이 얕아집니다.
- 잘못된 세션, 중복 제출, 빈 데이터, 권한 부족, Redis 장애 같은 케이스를 적습니다.

### 10. `테스트로 검증하기`

- 가장 중요한 테스트 파일과 검증 포인트를 적습니다.
- 가능하면 "왜 이 테스트가 핵심 리스크를 막는가"도 같이 적습니다.

### 11. `회고`

- 설계 선택의 장점과 남은 한계를 적습니다.

### 12. `취업 포인트`

- 1문장 답변
- 30초 답변
- 예상 꼬리 질문

### 13. `시작 상태`

- 이 글 전에 저장소가 어떤 상태인지 적습니다.
- 앞 글에서 이미 해결된 범위를 분명히 합니다.

### 14. `이번 글에서 바뀌는 파일`

- 이번 글에서 생성/수정되는 파일 범위를 다시 적습니다.
- 독자가 실제 구현할 때 체크리스트로 쓰게 합니다.

### 15. `구현 체크리스트`

- 구현 순서를 적습니다.
- 서로 의존하는 단계가 있으면 그 순서도 적습니다.

### 16. `실행 / 검증 명령`

- `./gradlew test`, `browserSmokeTest`, `publicUrlSmokeTest`, preflight script 등 실제 명령을 적습니다.

### 17. `산출물 체크리스트`

- 이 글이 끝나면 무엇이 가능해졌는지 체크박스처럼 확인하게 합니다.

### 18. `글 종료 체크포인트`

- 다음 글로 넘어가기 전에 독자가 설명할 수 있어야 하는 질문을 남깁니다.

### 19. `자주 막히는 지점`

- 초심자가 설계나 구현에서 흔히 틀리는 부분을 적습니다.
- WorldMap에서는 "프런트 책임과 서버 책임을 섞는 실수"를 특히 자주 경계합니다.

## 3. 재현성 기준

각 글은 아래 질문에 모두 `Yes`여야 합니다.

- 독자가 이 글만 보고 먼저 열 파일을 정할 수 있는가
- 독자가 구현 순서를 따라갈 수 있는가
- source of truth가 코드와 문서에서 일치하는가
- 검증 명령을 그대로 실행할 수 있는가
- 글이 끝났을 때 어떤 화면/API가 살아 있어야 하는지 아는가
- 실패 케이스를 읽고도 왜 이 설계가 필요한지 설명할 수 있는가
- 글이 커졌더라도 독자가 읽는 순서를 잃지 않는가

또한 아래 질문에도 `Yes`여야 합니다.

- 이 글이 문서만으로 닫히는 내용과, 실제 코드/테스트를 같이 열어야 닫히는 내용을 구분하는가
- 자동 테스트로 고정된 사실과 수동 운영 절차, 아직 남은 한계를 섞어 쓰지 않는가

## 4. 후반부 대표 글의 추가 규칙

`16`, `17`, `18`은 단순 기능 글이 아니라 production-ready bundle을 설명하는 대표 글입니다.

그래서 이 세 편은 아래를 추가로 만족해야 합니다.

- 표가 있어야 한다
  - runtime contract
  - verification lane
  - 테스트 맵
  - demo baseline
- “무엇을 의도적으로 안 하는가”도 적어야 한다
  - full browser E2E가 아닌 이유
  - request cache가 request 단위라는 점
  - required check와 repo workflow가 다른 레이어라는 점
- local 재현과 post-deploy 확인 순서를 분리해 적어야 한다
- 운영 설정과 app source of truth를 섞지 않아야 한다

## 5. 코드-문서 정합성 기준

문서는 코드보다 앞서 나가면 안 됩니다.

- 존재하지 않는 클래스나 메서드를 쓰지 않습니다.
- 현재 저장소 기준으로 삭제된 기능을 마치 살아 있는 것처럼 쓰지 않습니다.
- 과거 실험 흔적은 `legacy`, `rollback`, `현재 public scope 아님`처럼 명시합니다.
- 테스트 명령은 현재 통과 가능한 단위만 적습니다.
- `문서만 읽으면 다 된다`처럼 저장소보다 강한 약속을 하지 않습니다.
- 수동 운영 절차가 필요한 곳은 `코드로 고정`, `자동 검증`, `사람이 맞춰야 하는 단계`를 분리해서 적습니다.

WorldMap은 기능이 바뀌는 속도보다 설명 구조를 더 오래 유지해야 하므로, "옛날에는 이랬다"보다 "지금은 무엇이 source of truth인가"가 중요합니다.

## 6. 증명 범위 쓰는 법

특히 `16~18` 같은 후반부 대표 글은 아래 네 층을 섞어 쓰면 안 됩니다.

1. 코드로만 존재하는 사실
   - 예: 현재 profile 분기, 현재 controller/service 책임, 현재 DTO 구조
2. 테스트로 자동 고정된 사실
   - 예: 특정 integration test, browser smoke, config test가 보장하는 범위
3. 수동 운영 절차
   - 예: ALB health path 연결, GitHub repository variable 입력, 실제 배포 계정 wiring
4. 아직 없는 보장 또는 남은 한계
   - 예: production URL 실측 부재, full browser E2E 미도입, 특정 payload E2E 부재

좋은 글은 이 네 층을 한 문단 안에 뭉개지 않고, "무엇이 실제로 증명됐는가"를 계층별로 나눠 적습니다.

## 7. 검증 전략

WorldMap 블로그는 "테스트가 있다"를 추상적으로 말하지 않습니다.
검증은 아래 중 하나 이상을 반드시 포함합니다.

- unit test
- integration test
- controller/page test
- browser smoke
- public URL smoke
- script preflight

그리고 가능하면 아래까지 같이 적습니다.

- 어떤 리스크를 막는 테스트인가
- 어떤 실패 케이스를 대표하는가
- 독자가 직접 실행할 때 필요한 전제는 무엇인가

검증 섹션은 가능하면 아래 문장을 함께 포함해야 합니다.

- "이 테스트가 보장하지 않는 것은 무엇인가"
- "여기서부터는 수동 확인이 필요한가"

## 8. 글을 쪼개는 기준

글 하나에 너무 많은 문제를 넣으면 ERP 수준의 teaching quality가 나오지 않습니다.
WorldMap에서는 아래를 기준으로 글을 나눕니다.

- 도메인 경계가 다르면 글을 나눕니다
- 보는 사람이 다르면 글을 나눕니다
- runtime과 verification은 separate topic으로 다룹니다
- 반대로, 원인과 결과가 붙어 있어야 설명이 되는 내용은 하나로 묶습니다

예:

- `guest ownership`과 `simple auth`는 다른 문제라서 분리
- `mypage`와 `stats`는 같은 public read model 축이라서 묶기
- `dashboard`는 admin surface라서 따로 분리
- `scope reset`과 `new games lineup`은 public product boundary라는 같은 문제로 묶기

## 9. 권장 읽기 순서

### baseline

1. `01~04`

### data, game loop, leaderboard, recommendation

2. `05~10`

### ownership, auth, read model, admin surface

3. `11~14`

### public scope, runtime, hardening, verification

4. `15~18`

후반부 production-ready 파트만 먼저 읽고 싶다면 아래 묶음으로 읽는 편이 가장 좋습니다.

1. `16` 운영 런타임
2. `17` integrity / current member / current role
3. `18` verification / demo / interview pack

즉 `16 -> 17 -> 18`은 하나의 trilogy처럼 읽히도록 유지합니다.

## 10. 집필할 때 자주 생기는 실수

- 제목은 크지만 실제로는 두세 기능 로그를 한 파일에 붙여 놓는 것
- "왜" 없이 "무엇을 했다"만 나열하는 것
- 테스트 이름 없이 "테스트를 추가했다"고만 적는 것
- runtime 준비를 기능 코드와 같은 층으로 설명하는 것
- 권한과 ownership을 UI 조건문처럼 설명하는 것
- 현재 public scope 밖의 실험을 메인 흐름에 섞는 것
- 섹션 수를 억지로 줄이느라 runtime/hardening/verification 글의 표와 테스트 맵을 빼 버리는 것
- 자동 검증이 없는 운영 단계까지 "보장된다"라고 단정하는 것

## 11. 이 문서를 사용하는 순서

1. [README.md](./README.md)로 시리즈 구조를 봅니다.
2. 새 글을 쓸 때 이 문서의 19개 섹션을 먼저 복사해 골격을 만듭니다.
3. [00_quality_checklist.md](./00_quality_checklist.md)로 최종 점검합니다.
4. 글을 수정한 뒤에는 `README`, `PORTFOLIO_PLAYBOOK`, `WORKLOG`와 링크를 같이 확인합니다.

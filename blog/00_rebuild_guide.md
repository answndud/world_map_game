# Rebuild Guide

이 문서는 WorldMap 블로그 시리즈를 읽을 때 `무엇을 먼저 보고, 무엇을 직접 따라 해 봐야 하는지`를 정리한 가이드다.

## 이 시리즈를 읽는 법

이 시리즈는 단순 감상용이 아니라 `재현형 학습용`으로 읽는 것이 좋다.

현재 저장소 상태를 그대로 따라 만들고 싶다면,
번호 순으로 무조건 다 읽기보다 먼저 아래 글을 확인하는 편이 안전하다.

- [50-current-state-rebuild-map.md](./50-current-state-rebuild-map.md)

이 글은 예전 단계 기록과 현재 기준 구현 경로를 구분해 준다.

추천 순서는 아래와 같다.

1. 글 제목만 보지 말고 먼저 어떤 문제를 풀려는 글인지 확인한다.
2. "이번 글에서 다룰 파일" 섹션을 먼저 본다.
3. 코드 설명보다 "설계 구상"과 "실제 흐름"을 먼저 읽는다.
4. 마지막에 테스트와 취업 포인트를 확인한다.

## 글을 읽을 때 반드시 체크할 것

- 이 글이 해결하려는 문제는 무엇인가?
- 이 기능은 컨트롤러, 서비스, 엔티티 중 어디가 책임져야 하는가?
- 요청이 들어오면 어떤 객체와 상태가 바뀌는가?
- 실패 케이스는 무엇인가?
- 내가 면접에서 1분 안에 설명할 수 있는가?

## 작성자 체크포인트

글을 새로 쓰거나 업데이트할 때는 아래 항목이 있어야 한다.

- 실제 파일 경로
- 실제 클래스 이름
- 핵심 메서드 설명
- 요청 흐름 또는 데이터 흐름
- 최소 1개 이상의 테스트 근거
- 왜 이런 설계를 택했는지에 대한 이유

## 이 프로젝트에서 특히 중요한 관점

이 프로젝트는 게임 사이트처럼 보이지만, 설명할 때는 아래 관점이 더 중요하다.

- 서버가 게임 상태를 관리한다.
- 프론트는 입력과 표현을 담당한다.
- Redis는 단순 캐시가 아니라 랭킹 자료구조로 쓴다.
- AI는 서비스 런타임 호출이 아니라 설문 개선용 오프라인 도구로 제한한다.

즉, 화려한 UI보다 `설명 가능한 백엔드 구조`가 우선이다.

# [Spring Boot 포트폴리오] 01. 왜 WorldMap 게임 플랫폼을 주제로 잡았는가

## 1. 이번 글에서 풀 문제

Spring Boot 백엔드 포트폴리오를 만들 때 가장 흔한 문제는 "무난한 CRUD"에서 멈춘다는 점이다.

물론 CRUD도 중요하지만, 취업 포트폴리오에서는 아래 질문에 답할 수 있어야 한다.

- 서버가 직접 관리하는 상태가 있는가?
- 단순 게시판이 아니라 비즈니스 로직이 있는가?
- 성능이나 실시간성 관점에서 설명할 지점이 있는가?
- 면접에서 "왜 이 기술을 썼는가"를 말할 수 있는가?

WorldMap 프로젝트는 이 질문에 답하기 위해 시작한다.

이 글의 목표는 아래 세 가지다.

1. 왜 이 프로젝트를 "게임 사이트"가 아니라 "백엔드 중심 플랫폼"으로 봐야 하는지 설명한다.
2. 이 프로젝트가 취업 포트폴리오에서 어떤 강점을 가지는지 정리한다.
3. 이후 글에서 어떤 순서로 구현을 이어갈지 큰 흐름을 잡는다.

## 2. 먼저 알아둘 개념

### 2-1. 서버 주도 상태 관리

이 프로젝트에서 가장 중요한 개념이다.

예를 들어 위치 찾기 게임을 만든다고 해 보자.
프론트 화면에서 사용자가 지도를 클릭하더라도, 다음 정보는 서버가 책임져야 한다.

- 어떤 국가가 이번 문제인지
- 이미 몇 문제를 풀었는지
- 정답인지 아닌지
- 점수를 몇 점 줄지
- 게임이 끝났는지

즉, 프론트는 입력을 받고 화면을 보여 주지만, `게임의 진짜 상태`는 서버가 관리한다.

### 2-2. 도메인 로직

도메인 로직은 "이 서비스가 왜 존재하는가"를 보여 주는 핵심 규칙이다.

이 프로젝트에서는 아래가 도메인 로직이 된다.

- 국가 위치 정답 판정
- 거리 기반 점수 계산
- 인구수 오차율 계산
- 추천 설문 점수 계산
- 랭킹 반영 규칙

즉, 단순히 데이터를 저장하는 것이 아니라 `규칙을 코드로 만드는 일`이 중요하다.

### 2-3. 포트폴리오용 기술 선택

포트폴리오에서는 최신 기술을 많이 넣는 것보다, 기술 선택의 이유를 설명할 수 있어야 한다.

이 프로젝트의 기술 선택 이유는 비교적 명확하다.

- Spring Boot
  - 백엔드 서비스 구조와 계층 분리를 보여 주기 좋다.
- Thymeleaf
  - 프론트 복잡도를 낮추고 백엔드 로직 설명에 집중하기 좋다.
- Redis
  - 랭킹처럼 빠른 집계가 필요한 문제를 설명하기 좋다.
- LLM
  - 추천 결과의 설명을 자연어로 생성하는 데 쓸 수 있다.

## 4. 설계 구상

이 프로젝트는 처음부터 기능을 전부 한 번에 만들면 실패할 가능성이 높다.

예를 들어 아래 기능을 동시에 붙이기 시작하면 금방 복잡해진다.

- 지도 기반 위치 게임
- 인구수 퀴즈
- 실시간 랭킹
- AI 추천
- 회원 기능

그래서 설계를 할 때 가장 먼저 고정한 것은 `구현 순서`였다.

### 왜 구현 순서가 중요한가

초보자가 프로젝트를 만들 때 자주 겪는 문제는 "생각나는 기능을 그때그때 붙이는 것"이다.

이렇게 되면 나중에 아래 질문에 답하기 어려워진다.

- 왜 이 기능을 먼저 만들었는가?
- 앞 단계의 설계가 다음 단계에서 어떻게 재사용되는가?
- 어떤 코드가 공통 구조이고 어떤 코드가 기능별 로직인가?

그래서 WorldMap은 아래 순서로 간다.

1. 프로젝트 뼈대
2. 국가 데이터
3. 위치 찾기 게임 Level 1
4. 인구수 맞추기 게임 Level 1
5. Redis 랭킹
6. 설문 기반 추천
7. LLM 설명
8. 인증과 전적
9. Level 2 고도화

이 순서는 기술 나열이 아니라, `설명 가능한 성장 순서`다.

### 왜 위치 찾기 게임을 먼저 만드는가

이 프로젝트의 핵심은 서버가 게임 상태를 관리하는 구조다.

위치 찾기 게임은 그 구조를 가장 잘 보여 준다.

- 국가를 출제해야 한다.
- 중복 출제를 막아야 한다.
- 사용자의 좌표 입력을 받아야 한다.
- 정답 여부를 판정해야 한다.
- 점수를 계산해야 한다.
- 다음 문제로 넘어가야 한다.

즉, 한 기능 안에 세션, 라운드, 판정, 점수, 상태 전이가 다 들어 있다.

그래서 이 게임 하나를 제대로 만들면, 이후 인구수 게임과 랭킹도 더 쉽게 설명할 수 있다.

### 왜 AI를 마지막 쪽에 두는가

초보자 포트폴리오에서 자주 보이는 문제는 LLM 기능을 너무 일찍 붙여서, 오히려 핵심 백엔드 설명이 흐려지는 것이다.

이 프로젝트에서는 AI를 `설명 생성기`로 제한한다.

즉, 추천 결과를 정하는 것은 서버가 하고, LLM은 그 결과를 사람 친화적으로 풀어 준다.

이렇게 해야 아래 장점이 있다.

- 추천 결과의 근거를 설명할 수 있다.
- LLM 출력이 흔들려도 서비스 핵심 로직은 무너지지 않는다.
- 면접에서 "AI를 왜 그렇게 제한했는가"를 설계 관점으로 답할 수 있다.

## 5. 실제 흐름

아직 코드는 없지만, 앞으로의 요청 흐름은 대략 이렇게 잡는다.

### 5-1. 게임 흐름

1. 사용자가 게임 시작 버튼을 누른다.
2. 서버가 `game_session`을 만든다.
3. 서버가 현재 라운드 문제를 생성한다.
4. 사용자가 답을 제출한다.
5. 서버가 정답 판정과 점수 계산을 한다.
6. 서버가 라운드와 세션 상태를 갱신한다.
7. 게임 종료 시 결과를 저장하고 랭킹을 갱신한다.

### 5-2. 추천 흐름

1. 사용자가 설문에 답한다.
2. 서버가 점수를 계산해 상위 국가 후보를 정한다.
3. 서버가 결과를 저장한다.
4. 필요하면 LLM이 설명 문장을 생성한다.
5. 결과 페이지를 보여 준다.

핵심은 두 흐름 모두 `결정권이 서버에 있다`는 점이다.

## 6. 테스트로 검증하기

아직 구현 전 단계라서 테스트 코드는 없다.

대신 지금 시점에서 미리 정해야 하는 테스트 주제는 있다.

- 게임 정답 판정 테스트
- 점수 계산 테스트
- 세션 상태 전이 테스트
- Redis 랭킹 반영 테스트
- 추천 점수 계산 테스트

즉, 처음부터 "무엇을 테스트할 것인가"를 생각하고 설계해야 한다.

## 7. 회고

이 프로젝트를 시작하면서 가장 먼저 정리한 것은 기능이 아니라 `문서 구조와 작업 순서`였다.

이 선택은 초보자에게 특히 중요하다.
기능을 빨리 붙이는 것보다, 각 단계에서 왜 그런 선택을 했는지를 남겨야 나중에 면접에서 흔들리지 않기 때문이다.

이후 글에서는 실제로 Spring Boot 프로젝트 뼈대를 만들고, 어떤 의존성을 최소 단위로 넣을지 설명할 예정이다.

## 8. 취업 포인트

이 글에서 면접으로 연결될 수 있는 질문은 대략 아래와 같다.

- 왜 이 프로젝트를 CRUD가 아니라 게임형 플랫폼으로 잡았나요?
- 왜 서버 주도 상태 관리가 중요하다고 생각했나요?
- Redis는 어떤 기능에서 쓰려고 하나요?
- AI는 어떤 역할로 제한했나요?

짧게 답하면 이렇게 정리할 수 있다.

"이 프로젝트는 단순 CRUD보다 상태 관리, 점수 계산, 랭킹, 추천 같은 백엔드 비즈니스 로직을 보여 주기 좋았습니다. 특히 게임 진행 상태를 서버가 관리하도록 설계해서 서비스 계층의 책임과 도메인 모델링을 설명하기 좋고, Redis와 AI도 각각 랭킹과 설명 생성이라는 명확한 목적 아래 선택했습니다."

# [Spring Boot 포트폴리오] 05. 인구수 맞추기 게임 Level 1로 공통 세션 구조를 재사용하기

## 이번 글의 핵심 질문

이번 단계의 핵심은 단순히 새로운 게임 하나를 더 붙이는 것이 아니다.

질문은 이것이다.

“위치 게임에서 만든 구조를 어디까지 공통화하고, 어디부터 모드별로 남길 것인가?”

이번 글에서는 그 답을 `인구수 맞추기 게임 Level 1`으로 정리한다.

## 왜 두 번째 게임이 중요했는가

위치 게임 하나만 있으면 이런 구조가 나올 수 있다.

- 세션
- 라운드
- 답안 제출
- 결과 페이지

그런데 이 구조가 정말 좋은지 알려면 다른 종류의 게임을 하나 더 얹어봐야 한다.

그래야 아래가 보인다.

1. 진짜 공통인 것
2. 억지 공통화였던 것
3. 모드별로 달라야 하는 것

이번 단계는 바로 그 검증 단계다.

## 이번에 공통화한 것

이번 단계에서는 `game/common` 아래에 두 가지를 올렸다.

- `BaseGameSession`
- `GameSessionStatus`

왜 이 둘인가?

위치 게임과 인구수 게임 모두 세션 단위로 아래 정보를 똑같이 갖기 때문이다.

- 플레이어 닉네임
- 총 라운드 수
- 현재 라운드 번호
- 완료한 라운드 수
- 누적 점수
- 시작 시각
- 종료 시각
- 상태(`READY`, `IN_PROGRESS`, `FINISHED`)

즉, 세션의 “틀”은 같고 문제의 내용만 다르다.

그래서 세션 필드와 상태 전이만 공통 클래스로 올렸다.

## 왜 라운드는 공통화하지 않았는가

이건 오히려 중요한 판단이다.

위치 게임의 라운드는 이런 데이터를 가진다.

- 문제 국가 ISO3
- 사용자가 선택한 국가 ISO3
- 정답 여부
- 점수

반면 인구수 게임의 라운드는 이런 데이터를 가진다.

- 보기 4개
- 정답 보기 번호
- 사용자가 고른 보기 번호
- 정답 여부

즉, 라운드는 모양 자체가 다르다.

그래서 이번 단계에서는 라운드를 억지로 추상화하지 않았다.

이게 중요한 이유는 “공통점이 조금 있다고 해서 바로 상속이나 인터페이스로 묶지 않는다”는 판단을 보여주기 때문이다.

## 인구수 게임 Level 1을 어떻게 설계했는가

이번 버전은 보기형 문제다.

한 라운드의 흐름은 아래와 같다.

1. 서버가 나라 하나를 고른다.
2. 그 나라의 실제 인구수를 정답으로 잡는다.
3. 비슷한 규모의 국가 인구수를 주변 보기로 섞는다.
4. 총 4개의 보기 중 하나를 클라이언트에 내려준다.
5. 사용자는 보기 번호 하나를 고른다.
6. 서버가 정답 여부와 점수를 계산한다.

즉, 위치 게임의 “국가 선택 제출”이 여기서는 “보기 번호 제출”로 바뀐 것이다.

## 보기 생성은 어떻게 했는가

이번 단계의 핵심 클래스는 `PopulationGameOptionGenerator`다.

이 클래스는 아무 숫자 3개를 섞지 않는다.

대신:

1. 전체 국가를 인구수 기준으로 정렬한다.
2. 정답 국가 주변의 인구 규모를 우선 후보로 본다.
3. 그중에서 3개를 뽑아 정답과 함께 섞는다.

이렇게 하면 왜 좋은가?

- 문제가 너무 쉽거나 너무 말이 안 되게 되는 걸 줄일 수 있다.
- “보기 생성 규칙이 무엇인가?”라는 면접 질문에 답할 수 있다.
- 나중에 Level 2에서 수치 입력형으로 넘어가도 Level 1의 설계 이유가 남는다.

## 점수 계산은 왜 따로 뒀는가

인구수 게임도 `PopulationGameScoringPolicy`를 따로 뒀다.

현재 규칙은 단순하다.

- 맞으면 100점
- 틀리면 0점

단순해 보여도 정책 클래스로 분리한 이유는 분명하다.

1. 위치 게임처럼 모드별 판정 로직을 따로 두기 위해서
2. 나중에 부분 점수나 오답 패널티를 넣기 쉽게 만들기 위해서
3. 서비스가 “상태 조정”과 “비즈니스 규칙”을 동시에 떠안지 않게 하기 위해서

## 요청 흐름은 어떻게 바뀌었는가

위치 게임과 거의 같은 모양이다.

```mermaid
sequenceDiagram
    participant User
    participant API as PopulationGameApiController
    participant Service as PopulationGameService
    participant Option as PopulationGameOptionGenerator
    participant DB as H2/PostgreSQL

    User->>API: POST /api/games/population/sessions
    API->>Service: startGame()
    Service->>Option: 보기 4개 생성
    Service->>DB: session, rounds 저장
    User->>API: GET current round
    API->>Service: getCurrentRound()
    User->>API: POST answer(option number)
    API->>Service: submitAnswer()
    Service->>Service: 정답 여부, 점수 계산
    Service->>DB: round, session 갱신
    User->>API: GET session result
```

즉, 세션/라운드/결과 구조는 같고, 문제 생성 규칙과 판정 로직만 달라졌다.

## 이번 단계에서 배울 수 있는 설계 감각

이번 단계의 핵심은 “재사용” 자체가 아니다.

정확히는 “재사용할 것과 말 것을 구분하는 기준”이다.

재사용한 것:

- 세션 상태와 누적 점수 구조
- 시작/현재 라운드/답안/결과 API 패턴
- SSR + 바닐라 JS 셸 방식

재사용하지 않은 것:

- 라운드 엔티티
- 점수 정책
- 문제 생성 규칙

이 구분을 말로 설명할 수 있어야 진짜로 이해한 것이다.

## 테스트는 무엇을 확인했는가

### 1. 옵션 생성 단위 테스트

`PopulationGameOptionGeneratorTest`에서 확인한 것:

- 보기가 4개인가
- 정답 인구수가 포함되는가
- 중복 보기가 없는가

### 2. 전체 흐름 통합 테스트

`PopulationGameFlowIntegrationTest`에서 확인한 것:

- 세션 시작
- 현재 라운드 조회
- 정답 제출
- 마지막 라운드 후 종료
- 같은 라운드 중복 제출 방지

즉, “보기 생성 규칙”과 “실제 게임 상태 흐름”을 둘 다 테스트로 잡았다.

## 내가 꼭 답할 수 있어야 하는 질문

1. 왜 `BaseGameSession`만 공통화했는가?
2. 왜 라운드는 공통화하지 않았는가?
3. 보기 4개는 어떤 기준으로 만들었는가?
4. 왜 인구수 게임도 점수 정책 클래스를 따로 뒀는가?
5. 위치 게임과 인구수 게임의 공통점과 차이점은 무엇인가?

## 면접에서는 이렇게 설명하면 된다

“위치 게임을 먼저 만든 뒤, 두 번째 게임으로 인구수 맞추기 게임을 추가하면서 어떤 구조가 진짜 공통인지 검증했습니다. 세션 필드와 상태 전이는 `BaseGameSession`으로 재사용했고, 라운드와 점수 정책은 모드별로 다르기 때문에 따로 두었습니다. 인구수 Level 1은 보기형 문제로 시작했고, 보기 생성은 인구 규모가 비슷한 국가를 우선 섞는 방식으로 만들었습니다. 이렇게 해서 공통 구조와 모드별 로직의 경계를 설명 가능하게 만들었습니다.”

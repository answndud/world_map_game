# [Spring Boot 포트폴리오] 14. 오프라인 AI-assisted 설문 개선 루프 정리하기

## 이번 글의 핵심 질문

추천 기능에 AI를 쓰고 싶다면 가장 먼저 떠오르는 선택은 보통 “사용자 요청마다 LLM을 호출하자”다.

하지만 이 프로젝트에서는 그 방향을 택하지 않았다.

이번 글의 질문은 이것이다.

“AI를 서비스 런타임이 아니라 개발 단계의 설문 개선 도구로만 써도 충분하지 않을까?”

내 답은 `그렇다`였다.

## 왜 방향을 바꿨는가

런타임 LLM 호출은 분명 멋져 보인다.

하지만 현재 프로젝트 기준으로는 단점이 더 컸다.

1. 추천 결과를 볼 때마다 과금이 발생한다.
2. 같은 입력에 같은 결과가 나온다는 설명이 약해진다.
3. 추천 품질 문제가 생겼을 때 원인을 서버 규칙이 아니라 모델 응답 탓으로 돌리기 쉬워진다.

그래서 이 프로젝트는 방향을 바꿨다.

- 서비스 런타임:
  - 설문 답변을 서버가 계산
  - 결과는 deterministic
  - 만족도만 익명 수집
- 개발 단계:
  - AI가 문항 후보 생성
  - AI가 시나리오 생성
  - AI가 설문 개선 아이디어 비판

즉, AI를 “추천 설명 서비스”가 아니라 `오프라인 품질 개선 워크플로우`로 제한한 것이다.

## 런타임과 오프라인을 왜 분리했는가

이 프로젝트에서 추천 품질의 핵심은 “문장을 얼마나 그럴듯하게 쓰는가”가 아니다.

더 중요한 것은 아래다.

1. 설문 문항이 정말 좋은가
2. 선택지가 서로 헷갈리지 않는가
3. 특정 페르소나에서 결과가 한 지역으로 쏠리지 않는가
4. 실제 만족도 점수가 다음 버전 개선으로 이어지는가

이 문제들은 런타임 설명 생성보다 `설문 설계와 평가 자산`에 더 가깝다.

그래서 AI도 그쪽에만 쓰는 편이 더 합리적이었다.

## 어떤 서브 에이전트 역할을 두는가

현재 문서에서는 서브 에이전트 역할을 4개로 나눴다.

1. `survey-drafter`
   - 새 문항과 선택지 초안 생성
2. `survey-critic`
   - 모호하거나 중복된 문항 비판
3. `persona-generator`
   - 페르소나 평가 시나리오 생성
4. `eval-reviewer`
   - 실제 만족도 집계와 오프라인 시나리오 비교

중요한 점은, 이 에이전트들이 최종 결정을 내리는 것이 아니라는 점이다.

사람이 마지막에 버전 채택 여부를 결정한다.

## 페르소나 시나리오는 왜 필요한가

실제 사용자 만족도 데이터는 시간이 쌓여야 한다.

그 전에는 문항을 바꾸거나 점수식을 조정해도 바로 비교하기 어렵다.

그래서 `PERAONA_EVAL_SET` 같은 오프라인 시나리오 세트를 먼저 준비해 둔다.

예를 들어 아래 같은 시나리오를 넣는다.

- 저물가 + 음식 중심 + 도시 균형형
- 치안 + 복지 + 북유럽 기후형
- 영어 친화도 + 문화 다양성 + 빠른 도시 리듬형

이렇게 해야 “이 설문이 누구에게 잘 맞고, 누구에게는 어긋나는가”를 더 빨리 볼 수 있다.

## 왜 이 로직은 애플리케이션 코드보다 문서와 자산으로 먼저 잡는가

이번 단계는 서비스 기능을 더 넣는 작업이 아니다.

이미 서버 추천 엔진은 동작하고 있다.

지금 필요한 것은 그 엔진을 `어떻게 개선할지 반복 가능한 방식으로 설명하는 기준`이다.

그래서 먼저 문서와 시나리오 자산을 고정했다.

즉, 이 단계의 산출물은 새 API보다 `개선 루프` 그 자체다.

## 면접에서는 이렇게 설명하면 된다

“원래는 추천 결과 설명을 위해 런타임 LLM을 붙이는 안도 생각했지만, 과금과 비결정성 때문에 방향을 바꿨습니다. 대신 추천 계산은 서버가 deterministic하게 유지하고, AI는 개발 단계에서만 사용해 설문 문항 후보 생성, 페르소나 시나리오 생성, 개선안 비판을 수행하도록 했습니다. 그래서 서비스 안정성과 테스트 용이성은 유지하면서, 설문 품질 개선 속도만 AI의 장점으로 가져갔습니다.”

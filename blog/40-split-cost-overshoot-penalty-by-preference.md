# [Spring Boot 포트폴리오] 40. 비용 선호에 따라 초과 물가 패널티를 다르게 주기

## 왜 이 조각이 필요했는가

20문항 설문으로 확장한 뒤에도 약한 시나리오가 남아 있었다.

대표적인 경우가 `VALUE_FIRST` 성향인데, 생활비를 아끼고 싶다고 답해도 추천 결과에 상대적으로 비싼 국가가 너무 쉽게 남아 있었다.

즉, 문제는 질문 수가 아니라 `비용 선호를 점수식이 얼마나 진짜로 반영하느냐`였다.

이번 단계는 추천 엔진 전체를 다시 흔들지 않고, `초과 물가 패널티` 한 축만 더 정교하게 분리하는 실험이다.

## 기존 점수식의 한계

이전에는 초과 물가 패널티가 사실상 하나였다.

그래서 아래 세 응답이 모두 비슷하게 깎였다.

- 품질이 좋다면 어느 정도 비싸도 괜찮은 사람
- 적당한 균형을 원하는 사람
- 생활비를 확실히 아끼고 싶은 사람

이러면 `QUALITY_FIRST`와 `VALUE_FIRST`가 같은 나라를 너무 비슷하게 받게 된다.

## 이번에 바꾼 규칙

핵심은 단순하다.

- `VALUE_FIRST`
  - 실제 물가가 선호보다 높을 때 더 강하게 깎는다.
- `BALANCED`
  - 기존과 비슷한 정도로 깎는다.
- `QUALITY_FIRST`
  - 비싸더라도 바로 배제하지 않도록 더 약하게 깎는다.

코드는 `RecommendationSurveyService.costQualityPoints()` 안에서 처리한다.

즉, 요청 흐름은 그대로지만 서비스 내부 계산 규칙만 바뀐다.

## 요청 흐름은 어떻게 유지되는가

이번에도 추천 요청 자체는 그대로다.

1. 사용자가 `/recommendation/survey`에 들어간다.
2. `RecommendationQuestionCatalog`가 20개 질문을 렌더링한다.
3. `RecommendationSurveyForm`이 답변을 `RecommendationSurveyAnswers`로 변환한다.
4. `RecommendationSurveyService.recommend()`가 나라 프로필 30개를 점수화한다.
5. 결과 페이지는 top 3와 이유를 보여 준다.
6. 만족도는 `RecommendationFeedbackService`로 저장된다.

즉, 바뀐 것은 HTTP 구조가 아니라 `서비스 내부 점수 계산`이다.

## 왜 서비스가 맡아야 하는가

이 로직은 컨트롤러에 있으면 안 된다.

왜냐하면 아래 결정이 모두 추천 도메인 규칙이기 때문이다.

- 사용자의 비용 선호를 몇 단계로 볼 것인가
- 실제 물가가 한 단계 높을 때 몇 점을 깎을 것인가
- 같은 물가 차이라도 `VALUE_FIRST`와 `QUALITY_FIRST`를 다르게 볼 것인가

컨트롤러는 입력을 받아 타입으로 바꾸는 것까지만 해야 하고, 점수 계산은 `RecommendationSurveyService`가 맡아야 구조를 설명하기 쉽다.

## 무엇이 실제로 달라졌는가

`engine-v5` 기준 snapshot을 다시 고정해 보니 변화는 아래처럼 나타났다.

- `P02`
  - `태국, 스페인, 이탈리아`
  - ->
  - `태국, 스페인, 말레이시아`
- `P14`
  - `스페인, 태국, 이탈리아`
  - ->
  - `스페인, 태국, 말레이시아`
- `P15`
  - `포르투갈, 뉴질랜드, 우루과이`
  - ->
  - `포르투갈, 뉴질랜드, 남아프리카 공화국`

즉, 저비용 시나리오의 후보 구성이 실제로 움직였다.

특히 `P02`, `P14`는 더 저비용 쪽 후보가 3위로 올라왔다.

반면 `P15`는 아직 원하는 방향으로 완전히 정리되지 않았다. 그래서 이 시나리오는 다음 튜닝에서도 계속 약한 시나리오로 볼 예정이다.

## 테스트는 무엇을 봤는가

이번 조각에서는 세 가지를 같이 봤다.

1. `RecommendationOfflinePersonaSnapshotTest`
   - `engine-v5`의 18개 시나리오 top 3를 새 기준으로 다시 고정
2. `RecommendationOfflinePersonaCoverageTest`
   - baseline 하한이 무너지지 않는지 확인
3. 통합 테스트
   - `RecommendationPageIntegrationTest`
   - `RecommendationFeedbackIntegrationTest`
   - `AdminPageIntegrationTest`

즉, 점수식만 바꾼 것이 아니라 `결과, 피드백, 운영 화면 버전 표기`까지 같이 맞췄다.

## 이번 단계의 의미

이 조각의 의미는 “추천 엔진을 완성했다”가 아니다.

오히려 반대다.

- 큰 설계는 유지한다.
- 약한 시나리오를 하나씩 겨냥한다.
- 어떤 축을 왜 조정했는지 남긴다.
- 결과는 snapshot으로 다시 고정한다.

이 방식이 있어야 나중에 추천 엔진이 왜 바뀌었는지 설명할 수 있다.

## 면접에서는 이렇게 설명하면 된다

“20문항 설문까지 만든 뒤에도 비용 민감 시나리오가 충분히 구분되지 않았습니다. 그래서 이번에는 전체 추천 알고리즘을 다시 짜지 않고, 초과 물가 패널티만 사용자의 비용 선호에 따라 다르게 주는 실험을 했습니다. 이 계산은 `RecommendationSurveyService`가 맡고, 결과는 18개 페르소나 snapshot을 다시 고정해서 어떤 시나리오가 어떻게 바뀌었는지 추적 가능하게 만들었습니다.”

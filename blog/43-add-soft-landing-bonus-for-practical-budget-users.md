# 43. 현실형 저예산 사용자에게 soft landing bonus 추가하기

## 왜 이 조각이 필요했나

`engine-v7`까지 오면서 `P04`와 `P15`는 꽤 좋아졌지만, `P06`은 아직 `스페인, 우루과이, 이탈리아`가 top 3였다.

이 시나리오는 단순히 “물가가 낮으면 좋다”가 아니라,

- 생활비 압박은 낮아야 하고
- 치안도 중요하고
- 도시 편의와 숨 쉴 공간이 둘 다 필요하고
- 영어 적응도 아주 무시할 수는 없는

현실형 사용자였다.

즉 음식이나 문화 점수보다, 실제로 처음 가서 덜 막히는 나라가 더 맞아야 했다.

## 먼저 gap을 확인했다

이번에는 바로 점수식을 더 바꾸지 않았다.

먼저 `P06` 한 케이스를 디버그 테스트로 직접 찍어 실제 점수를 확인했다.

```text
스페인   314
우루과이 298
이탈리아 293
포르투갈 287
```

차이는 6점뿐이었다.

즉 전체 추천 엔진을 다시 갈아엎을 필요는 없고, `포르투갈`만 아주 좁게 밀어 올리면 되는 상태였다.

## 어떤 bonus를 추가했나

이번에는 `softLandingBonus()`를 추가했다.

이 bonus는 모든 시나리오에서 켜지지 않는다.

- `VALUE_FIRST`
- `SAFETY HIGH`
- `MIXED`
- `BALANCED`
- `EnglishSupport MEDIUM`

이 다섯 조건이 모두 맞을 때만 켜진다.

그리고 나라 프로필도 아래 조건을 동시에 만족해야 한다.

- `englishSupport >= 4`
- `newcomerFriendliness >= 4`
- `housingSpace >= 4`
- `safety >= 4`
- `welfare >= 4`
- `priceLevel < 4`

즉, “너무 비싸지 않으면서도, 영어로 적응하고 정착 기반을 만들기 쉬운가”를 별도 신호로 본 것이다.

## 왜 서비스 로직에 넣었나

이 규칙은 컨트롤러가 아니라 `RecommendationSurveyService`에 있어야 한다.

이유는 간단하다.

- 어떤 응답 조합에서 bonus를 켤지
- 어떤 나라 속성을 함께 읽을지
- 몇 점을 줄지

이 전부가 HTTP 바인딩 문제가 아니라 추천 도메인 규칙이기 때문이다.

## 결과는 어떻게 바뀌었나

이제 `P06`의 top 3는 이렇게 바뀌었다.

```text
스페인, 우루과이, 포르투갈
```

즉 3위 후보가 `이탈리아`에서 `포르투갈`로 교체됐다.

이번 조각은 전체 엔진을 다시 흔들지 않고, 실제 gap을 확인한 뒤 아주 좁은 신호 하나만 추가한 사례다.

## 테스트

다음 기준으로 확인했다.

1. 디버그 테스트로 `P06` 후보 점수 직접 확인
2. `RecommendationOfflinePersonaSnapshotTest`에서 `engine-v8` snapshot 재고정
3. `RecommendationOfflinePersonaCoverageTest`에서
   - `P06`이 `우루과이`, `포르투갈`을 포함하는지
   - `이탈리아`는 빠지는지
4. 추천 관련 targeted suite 통과
5. `./gradlew test` 전체 통과

## 면접에서 어떻게 설명할까

“추천 엔진을 튜닝할 때는 전체 알고리즘을 자주 갈아엎지 않았습니다. 먼저 weak scenario 하나를 정하고, 실제 후보 점수를 직접 확인한 뒤 필요한 축 하나만 좁게 조정했습니다. 이번에는 `P06`의 `이탈리아 -> 포르투갈` 교체가 목표였고, 그래서 `softLandingBonus`를 추가해 영어 적응, newcomer 친화도, 주거 안정성이 모두 높은 저비용 후보를 조금 더 올렸습니다.”

# 다음에 어떤 게임을 더 넣을지 먼저 설계하기

## 왜 이 조각이 필요했는가

현재 WorldMap은

- 국가 위치 찾기
- 국가 인구수 맞추기
- 나라 추천
- 랭킹

까지는 안정화돼 있다.

다음 단계에서 자연스러운 질문은 이것이다.

`이 테마에서 어떤 게임을 더 붙일 수 있을까?`

하지만 여기서 중요한 건 단순히 아이디어를 많이 내는 게 아니다.

- 현재 데이터로 바로 가능한가
- 기존 도메인 구조를 재사용할 수 있는가
- 면접에서 왜 이 순서로 추가했는지 설명 가능한가

를 같이 봐야 한다.

## 이번에 고른 세 가지

1. 수도 맞히기
2. 인구 비교 퀵 배틀
3. 국기 보고 나라 맞히기

## 왜 수도 맞히기가 먼저인가

이유는 단순하다.

- `country.capitalCity`가 이미 있다.
- 4지선다 보기형 구조가 익숙하다.
- 기존 인구수 게임처럼 `문제 생성 -> 보기 생성 -> 정답 판정 -> 점수/하트`를 서버가 그대로 가져갈 수 있다.

즉, 새 데이터 파이프라인 없이 가장 빨리 여는 새 게임이다.

## 왜 인구 비교 퀵 배틀이 두 번째인가

현재 `country.population`도 이미 있다.

그래서 두 나라를 보여주고
`인구가 더 많은 나라를 고르기`
같은 비교형 게임은 구현이 쉽다.

다만 현재 인구수 게임과 너무 비슷해질 수 있으므로,
수도 맞히기보다 먼저 열 이유는 약하다.

즉, 두 번째가 적절하다.

## 왜 국기 게임은 세 번째인가

국기 게임은 사용자 반응은 좋을 가능성이 크다.

하지만 실제로는

- 국기 자산 source
- 정적 파일 경로 정책
- local/demo 포함 방식

같은 준비가 먼저 필요하다.

즉, 이 게임의 핵심은 규칙보다 에셋 파이프라인이다.

그래서 세 번째가 맞다.

## 현재 구조와 어떻게 연결되는가

핵심 원칙은 변하지 않는다.

새 게임도 모두

- session
- stage
- attempt
- leaderboard_record

기준으로 설명 가능해야 한다.

즉,
프론트는 입력과 표현,
서버는 문제 생성, 정답 판정, 점수 계산, 하트 감소, 다음 Stage 생성을 맡는다.

## 첫 구현 조각은 무엇인가

다음 코딩 조각은 `수도 맞히기 Level 1 vertical slice`다.

여기서 먼저 만들 것은 아래다.

1. `capital` game mode
2. start/state/answer/result 흐름
3. 4지선다 보기 생성
4. 최소 1개 통합 테스트
5. 랭킹 반영

## 면접에서는 어떻게 설명할 것인가

> 새 게임을 추가할 때는 재미있는 아이디어보다 현재 데이터와 구조를 얼마나 재사용할 수 있는지를 먼저 봤습니다. 그래서 `capitalCity`가 이미 있는 수도 맞히기를 첫 확장 대상으로 잡고, 그 다음 population 재사용이 쉬운 인구 비교 배틀, 마지막으로 에셋 파이프라인이 필요한 국기 게임 순으로 열기로 했습니다. 이렇게 하면 기능을 늘리면서도 왜 이 순서인지 설명할 수 있습니다.

# Quality Checklist

## 1. 사용 방법

이 문서는 WorldMap 블로그 글이 아래 네 기준을 만족하는지 점검하기 위한 체크리스트입니다.

- 초보자 이해도
- 재현성
- 코드-문서 정합성
- 면접 대응력

각 글을 수정한 뒤에는 이 문서를 다시 확인합니다.

## 2. A. 초보자 이해도

### A-1. 문제 정의가 먼저 보이는가

- 글 첫머리에서 "무엇을 만들었다"가 아니라 "무슨 문제를 풀었는가"가 먼저 보이는가
- 글 제목만 보고도 어떤 설계 문제를 다루는지 감이 오는가

### A-2. 용어가 설명되는가

- `Session`, `Stage`, `Attempt`, `guest ownership`, `read model`, `terminal result` 같은 표현을 바로 던지지 않는가
- 적어도 한 번은 평이한 말로 풀어 썼는가

### A-3. 파일이 먼저 보이는가

- 독자가 어떤 파일부터 열어야 하는지 바로 알 수 있는가
- controller/service/domain/template/test가 균형 있게 소개되는가

### A-4. 글의 중심이 한 문제에 고정되는가

- 글 하나가 두세 주제를 억지로 떠안고 있지 않은가
- 다른 독자층의 화면을 한 글에 섞지 않았는가

## 3. B. 재현성

### B-1. 시작 상태가 적혀 있는가

- 이 글을 읽기 전에 어떤 기능이 이미 존재하는지 적혀 있는가
- 앞 글과 현재 글의 경계가 보이는가

### B-2. 최종 도착 상태가 적혀 있는가

- 글이 끝났을 때 어떤 URL/API/화면이 살아 있어야 하는가
- 어떤 상태 전이를 보여 줄 수 있어야 하는가

### B-3. 구현 순서가 적혀 있는가

- 구현 체크리스트가 단순 TODO가 아니라 실제 의존 순서를 반영하는가
- 무엇을 먼저 만들면 안 되는지도 보이는가

### B-4. 검증 명령이 있는가

- 독자가 그대로 실행할 수 있는 명령이 있는가
- test, browser smoke, script preflight 같은 실제 실행 단위가 적혀 있는가

### B-5. 실패 케이스가 적혀 있는가

- 잘못된 세션 접근, stale submit, Redis 장애, 권한 부족 같은 실패 케이스를 적었는가
- "정상 흐름"만 나열하고 끝나지 않았는가

### B-6. 긴 글이어도 읽기 순서가 보이는가

- 섹션이 많아져도 지금 어디를 읽고 있는지 잃지 않는가
- 특히 `16~18`처럼 긴 글은 표, 실행 순서, 테스트 맵으로 길을 잡아 주는가

### B-7. 재현 약속이 과장되지 않는가

- 이 글이 문서만으로 끝나는 것처럼 과장하지 않는가
- 실제로는 링크된 코드, 테스트, 설정 파일을 함께 봐야 한다는 점이 드러나는가
- 자동 검증과 수동 운영 절차가 섞여 있지 않은가

## 4. C. 코드-문서 정합성

### C-1. 실제 파일을 가리키는가

- 링크가 전부 현재 저장소의 실제 파일을 가리키는가
- 존재하지 않는 클래스나 과거 파일 이름을 쓰지 않았는가

### C-2. 현재 제품 범위를 반영하는가

- 현재 public scope 밖의 실험을 메인 기능처럼 쓰지 않았는가
- legacy/rollback 범위는 명시적으로 구분했는가

### C-3. source of truth가 명확한가

- 상태 변경이 어디서 일어나는지 분명한가
- 컨트롤러가 아니라 서비스나 도메인에 있어야 하는 이유가 적혀 있는가

### C-4. 테스트가 현재 상태와 맞는가

- 실행 명령이 현재 통과 가능한 테스트를 가리키는가
- 왜 그 테스트가 핵심 리스크를 막는지 설명하는가

### C-5. 증명 범위를 정확히 말하는가

- 코드로 존재하는 사실, 테스트로 고정된 사실, 사람이 수동으로 해야 하는 절차를 구분하는가
- local fallback 결과를 production 근거처럼 과장하지 않는가
- fixture 기반 검증과 live 환경 사실을 같은 수준으로 말하지 않는가

## 5. D. 면접 대응력

### D-1. 1문장 답변이 있는가

- 이 기능을 1문장으로 요약할 수 있는가

### D-2. 30초 답변이 있는가

- 클래스 이름, 상태 변화, 설계 이유를 섞어 30초 안에 설명할 수 있는가

### D-3. 예상 꼬리 질문이 있는가

- "왜 프런트가 아니라 서버인가"
- "왜 Redis와 DB를 같이 두는가"
- "왜 current role을 다시 보는가"
- "왜 browser smoke를 따로 두는가"

같은 질문에 대비할 수 있게 돼 있는가

## 6. E. 연재 흐름

### E-1. 이전 글과 다음 글이 자연스럽게 이어지는가

- 지금 글이 왜 이 시점에 필요한지 보이는가
- 다음 글에서 어떤 문제를 열지 자연스럽게 연결되는가

### E-2. 프로젝트 전체 순서를 해치지 않는가

- baseline보다 운영 hardening을 먼저 설명하지 않는가
- 공통 데이터 없이 게임 구현부터 들어가지 않는가
- 런타임과 verification을 기능 코드보다 먼저 전면에 두지 않는가

### E-3. production-ready trilogy가 번들처럼 읽히는가

- `16 -> 17 -> 18`이 runtime, hardening, verification 순서로 자연스럽게 이어지는가
- 세 글이 서로 같은 단어를 다른 뜻으로 쓰지 않는가
- GitHub workflow, required check, browser smoke, public URL smoke, demo bootstrap의 역할이 일관되게 설명되는가
- 각 글이 "무엇이 실제로 보장되는가"를 같은 톤으로 쓰는가

## 7. F. 문체와 용어

### F-1. 과장보다 구조가 앞서는가

- "멋지다", "혁신적이다" 같은 과장을 줄이고 실제 구조를 설명하는가

### F-2. 불필요한 추상화 언어를 피하는가

- 쉽게 풀어 쓸 수 있는 내용을 괜히 프레임워크 용어로 포장하지 않는가

### F-3. 문장이 너무 짧거나 메모처럼 끊기지 않는가

- 핵심 섹션은 적어도 문단 단위로 설명되는가
- bullets만 나열하고 끝나지 않는가

## 8. G. 현재 구현의 한계 고지

- 남아 있는 한계를 숨기지 않는가
- production URL 미배포, social login 미지원, flag 자산 수 제한 등 현재 한계를 적는가
- 한계를 적되 "그래서 지금 구조가 왜 여전히 설명 가치가 있는가"도 함께 적는가
- 후반부 글이라면 `자동 검증 없음`, `수동 배포 필요`, `live URL 미확보` 같은 한계를 숨기지 않는가

## 9. H. 글별 완료 기준

글 하나가 아래를 모두 만족하면 완료로 봅니다.

- 제목만 보고 어떤 문제를 닫는지 알 수 있다
- 본문이 `문제 -> 설계 -> 코드 -> 흐름 -> 테스트 -> 회고`를 따른다
- 실제 파일 링크가 충분하다
- 실패 케이스가 적혀 있다
- 검증 명령이 있다
- 취업 포인트가 있다
- 구현 체크리스트와 산출물 체크리스트가 있다

## 10. 우선 적용 순서

1. 허브 문서가 최신 구조를 반영하는지 확인
2. 본편에 `최종 도착 상태`, `핵심 도메인 모델 / 상태`, `실패 케이스 / 예외 처리`가 있는지 확인
3. 테스트 파일과 실행 명령이 실제와 맞는지 확인
4. 면접 답변과 꼬리 질문이 있는지 확인

## 11. 빠른 점수표

아래 항목을 각 0~2점으로 간단히 채점할 수 있습니다.

- 문제 정의
- 재현성
- 코드-문서 정합성
- 실패 케이스 설명
- 테스트/검증
- 증명 범위의 정직함
- 면접 대응력
- 연재 흐름
- 긴 글의 길 찾기

총점 기준:

- `12~14`: 공개 가능
- `9~11`: 보강 필요
- `8 이하`: 구조부터 다시 써야 함
